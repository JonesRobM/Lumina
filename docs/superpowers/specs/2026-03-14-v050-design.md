## Feature 1 — 1D Bessel Spectral Sum

### Problem

`EwaldGreens::recip_space()` returns a zero tensor for `LatticeSpec::Chain` (1D periodic BC). The real-space direct sum with 20 shells (~2% accuracy) is the only contribution. For unit cells where dipoles have non-zero transverse separation ρ > 0 (e.g. a 1D periodic dimer or helical chain), the reciprocal-space spectral sum is needed for correct far-field behaviour.

### Physics

For a 1D chain periodic in x with Bloch factor exp(ik_Bx·x) and period a₁, Poisson summation gives:

```
G_recip(Δr) = (1/2πa₁) Σ_m  exp(i G_m Δx) · T(G_m, ρ, κ_m)
```

where:
- G_m = k_Bx + 2πm/a₁ (reciprocal lattice vector)
- κ_m = csqrt_positive_imag(G_m² − k²)  (Im ≥ 0 branch — already in codebase)
- ρ = √(Δy² + Δz²)  (transverse separation)
- T is a 3×3 tensor assembled from K₀(κ_m ρ) and K₁(κ_m ρ)/ρ via ∇∇ applied to the scalar sum

The tensor components follow from applying ∂ᵢ∂ⱼ to the scalar spectral sum:
- Longitudinal (xx): −G_m² terms + k² contribution
- Transverse (yy, zz) and mixed (xy, xz, yz): K₁(κ_m ρ)/ρ and K₀ terms weighted by Δy/ρ, Δz/ρ

### Mode dispatch

| Mode | κ_m | Bessel functions used |
|------|-----|----------------------|
| Evanescent (G_m > k) | real positive β | K₀(βρ), K₁(βρ) — real-valued |
| Propagating (G_m < k) | purely imaginary iα (α > 0) | K₀(iαρ), K₁(iαρ) — see identities below |

**Propagating-mode Bessel identities** (A&S 9.6.4, `K_ν(iz) = (π/2)i^{ν+1} H_ν^{(1)}(z)`):

```
K₀(iαρ) = (π/2)(−Y₀(αρ) + i J₀(αρ))
K₁(iαρ) = (π/2)(−J₁(αρ) − i Y₁(αρ))   [i.e. −(π/2)(J₁ + iY₁)]
```

These follow from H_ν^{(1)}(z) = J_ν(z) + iY_ν(z) and i^1 = i, i^2 = −1. The test
`test_k0_complex_imaginary` must verify the `+i J₀` sign for the imaginary part explicitly
(the formula has been corrected from an earlier erroneous `−i J₀` version).

J₀, J₁ already available in `sommerfeld.rs` — will be re-implemented locally in `ewald.rs`
(avoids a cross-module dependency on the solver::sommerfeld private namespace).

**Scope note**: In this codebase `k` is always real (wavenumber k = 2πn_env/λ, n_env real). Complex dielectric functions enter through the polarisability tensor, not through k. Therefore κ_m = csqrt_positive_imag(G_m² − k²) is always either purely real (evanescent) or purely imaginary (propagating), and the two-branch dispatch in `bessel_k0_cplx`/`bessel_k1_cplx` is complete. General complex-argument K₀/K₁ (needed only for complex k, e.g. absorbing background media) is deferred.

### ρ = 0 guard

For a strictly collinear 1D chain (all dipoles at y = z = 0), ρ = 0. The spectral sum diverges term-by-term because K₀(κ_m · 0) = ∞. The divergences cancel with the real-space sum across the full Ewald identity, but evaluating them separately is numerically unstable.

Guard: `if ρ < 1e-10 { return zero_tensor; }`.

**Why this is exact for the collinear case**: For strictly collinear dipoles (ρ = 0, Δr = (Δx, 0, 0)), the spectral sum's transverse tensor components (yy, zz, xy, xz, yz) all contain `K₁(κ_m ρ)/ρ` factors weighted by `Δy/ρ` or `Δz/ρ`. By L'Hôpital or series expansion, `K₁(κρ)/ρ · (Δy/ρ)` → 0 as ρ → 0 when Δy = 0. The longitudinal (xx) spectral contribution reduces to a pure phase sum `Σ_m e^{iG_m Δx} (−k²/κ_m²) K₀(κ_m ρ)` which, at ρ → 0, diverges — but this divergence is exactly cancelled by the real-space sum's convergent direct value. Since the combined Ewald result must equal the direct per-shell sum (which converges for Δx ≠ 0), returning zero for the spectral part and relying on the 20-shell real-space sum is justified: for a collinear chain the spectral sum contributes zero net correction to all components after cancellation.

In practice the guard threshold of 1e-10 nm covers all physically realistic collinear dipole positions.

**Accuracy note**: When the ρ=0 guard fires, the caller relies on the 20-shell real-space direct sum only — the same ~2% accuracy as the pre-v0.5.0 1D Ewald implementation. The Bessel spectral sum does not improve accuracy for strictly collinear chains. Users who need higher accuracy for collinear 1D chains should increase the shell count in `real_space` or use a finer real-space sum; the Bessel path does not help here.

### Implementation notes

- Shell count M = min(⌈k·a₁/2π⌉ + 10, 50) each side — ensures |κ_m|·ρ ≥ 10 for outermost evanescent term; cap of 50 prevents runaway cost for dense short-wavelength chains (k·a₁/2π ≈ 0.02 at 500 nm for a₁ = 10 nm, so in practice M ≈ 10–20)
- `bessel_k0`/`bessel_k1`: piecewise A&S 9.8.5–9.8.8 polynomial rational approximations, accuracy < 1.5×10⁻⁷ (consistent with existing `erfc_real`)
- `bessel_y0`/`bessel_y1`: piecewise A&S 9.4.1–9.4.6, same accuracy target

## Feature 2 — FFT-Accelerated Matvec (Block-Toeplitz)

### Problem

`matvec_on_the_fly` is O(N²) per GMRES iteration. For large nanoparticles on a regular cubic grid (the typical output of `discretise_primitive`), the interaction matrix is 3-level block-Toeplitz — the off-diagonal blocks depend only on the displacement r_i − r_j. A 3D FFT convolution reduces the matvec to O(N log N).

### Activation condition

All four conditions must hold:
1. `N > iterative_threshold` (GMRES path active)
2. `solver.lattice.is_none()` — no periodic boundary conditions
3. `params.substrate_runtime.is_none()` — no substrate (breaks translation invariance in z)
4. `detect_regular_grid(dipoles, cell_size).is_some()` — dipoles on a regular cubic grid

Falls back to `matvec_on_the_fly` silently if any condition fails.

### Regular lattice detection

```rust
fn detect_regular_grid(dipoles: &[Dipole], cell_size: f64)
    -> Option<(usize, usize, usize)>
```

Algorithm:
1. Compute bounding box; grid dimensions `Nx = (x_max − x_min)/cell_size + 1` (rounded)
2. For each dipole position, check `|(pos[α] − box_min[α]) / cell_size − round(...)| < 1e-6`
3. Return `Some((Nx, Ny, Nz))` only if all dipoles pass; `None` otherwise

### `FftMatvecPlan` struct (new file `cda/fft_matvec.rs`)

```rust
pub struct FftMatvecPlan {
    nx: usize, ny: usize, nz: usize,
    /// Flat index for each dipole in the (2Nx × 2Ny × 2Nz) padded grid
    grid_index: Vec<usize>,
    /// Precomputed FFT of 9 kernel components G_αβ (α,β ∈ {0,1,2})
    /// Shape: 9 × (2Nx · 2Ny · 2Nz)
    kernel_fft: Vec<Vec<Complex64>>,
    fft_len: usize,   // 2Nx · 2Ny · 2Nz
}
```

```rust
impl FftMatvecPlan {
    pub fn try_build(dipoles: &[Dipole], cell_size: f64, k: f64)
        -> Option<Self>;
    pub fn matvec(&self, dipoles: &[Dipole], x: ArrayView1<Complex64>)
        -> Array1<Complex64>;
}
```

### Kernel precomputation

For each component pair (α, β) ∈ 0..9:
1. Allocate a `(2Nx × 2Ny × 2Nz)` complex buffer, initialised to zero
2. Loop bounds: `Δl ∈ [−(Nx−1), Nx−1]`, `Δm ∈ [−(Ny−1), Ny−1]`, `Δn ∈ [−(Nz−1), Nz−1]`, skip (0,0,0):
   - `r = [Δl·d, Δm·d, Δn·d]`; compute G_αβ(r) via `dyadic_greens_tensor`
   - Circulant index (using **signed** integer `rem_euclid`, not `%`):
     ```rust
     let cl = Δl.rem_euclid(2 * Nx as isize) as usize;
     let cm = Δm.rem_euclid(2 * Ny as isize) as usize;
     let cn = Δn.rem_euclid(2 * Nz as isize) as usize;
     buf[cl * (2*Ny*2*Nz) + cm * 2*Nz + cn] = G_αβ;
     ```
   - Note: Rust's `%` on signed integers is remainder (can be negative); `rem_euclid` is required to map negative displacements to the upper half of the buffer correctly.
3. Run 3D FFT (three passes of 1D FFTs along x, y, z axes using `rustfft`) and store

The G(0,0,0) slot remains zero (zero-initialised buffer, (0,0,0) skipped in the loop) — self-interaction excluded.

### Per-matvec computation

The `matvec` method signature accepts `dipoles: &[Dipole]` so it can recompute `invert_3x3(alpha_i)` per dipole for the diagonal — O(N) and negligible vs the FFT work.

1. Diagonal: `y[3i..3i+3] += invert_3x3(dipoles[i].alpha) · x[3i..3i+3]`
2. For β ∈ {0,1,2}:
   a. Scatter `x_β` components onto the `(2Nx × 2Ny × 2Nz)` padded grid
   b. Forward FFT3
   c. For α ∈ {0,1,2}: pointwise multiply with `kernel_fft[α*3+β]`, add to accumulator
3. Inverse FFT3 for each α accumulator; gather `y_α` from grid positions

## Feature 3 — GPU Matrix Assembly Shader

### Problem

Currently, the full 3N×3N interaction matrix is assembled CPU-side by Rayon (O(N²) Green's function evaluations) and then uploaded to the GPU. For large N, the CPU assembly time dominates. A GPU compute shader can evaluate all (i,j) Green's tensor blocks in parallel, eliminating the CPU bottleneck and the PCIe upload of the fully-assembled matrix.

### Shader: `matrix_assemble.wgsl`

Dispatch: `⌈3N/16⌉ × ⌈3N/16⌉` workgroups, 16×16 threads per group — one thread per (row, col) in the 3N×3N matrix.

Per-thread:
1. `i = row / 3`, `j = col / 3`, `α = row % 3`, `β = col % 3`
2. If `i == j`: write `(0.0, 0.0)` (diagonal block left as zero; α⁻¹ patched separately)
3. Otherwise: compute `R = positions[i] - positions[j]`, evaluate `G_αβ(R, k)` in f32, write to `matrix[row * dim + col]`

Bindings:
```wgsl
@group(0) @binding(0) var<storage, read>       positions : array<vec4<f32>>;
@group(0) @binding(1) var<storage, read_write>  matrix   : array<vec2<f32>>;
@group(0) @binding(2) var<uniform>              params   : Params;
// Params { dim: u32, n_dipoles: u32, k_re: f32, k_im: f32 }
```

The free-space dyadic G expression is identical to `dyadic_greens_tensor` in greens.rs, translated to f32 WGSL. The f32 precision floor (~1e-7) is consistent with the existing GPU matvec shader.

### New helper in `assembly.rs`

The diagonal blocks (α⁻¹ per dipole) are currently computed inside `assemble_interaction_matrix`. A new public function must extract them independently. Diagonal blocks are pure algebraic inversions of each dipole's polarisability tensor — they do not depend on k, use_fcd, or cell_size (those only affect off-diagonal Green's function terms):

```rust
/// Compute the 3×3 inverse polarisability block for each dipole.
/// Returns a Vec of length N, each a column-major [Complex64; 9].
/// This is a pure inversion of each dipole's pre-built polarisability tensor;
/// no k, FCD, or cell_size dependency.
pub fn compute_diagonal_blocks(dipoles: &[Dipole]) -> Vec<[Complex64; 9]>
```

This calls `invert_3x3_pub(&dipole.polarisability)` for each dipole — the same path already used inside `assemble_interaction_matrix`.

### New trait method on `ComputeBackend`

Default implementation returns `None` (CPU backend unchanged). Note `k: f64` — consistent with all existing solver functions (wavenumber is always real in this codebase):

```rust
fn assemble_and_create_session(
    &self,
    positions: &[[f64; 3]],
    k: f64,
    diagonal_blocks: &[[Complex64; 9]],  // output of compute_diagonal_blocks(dipoles)
) -> Option<Result<Box<dyn MatvecSession>, ComputeError>> {
    None
}
```

GPU implementation:
1. Convert positions to f32 (x,y,z,0.0 padding → `vec4<f32>`)
2. Call `new_session(3*n)` to allocate the 5 pre-sized GPU buffers
3. Dispatch `matrix_assemble.wgsl` (fills off-diagonal blocks in f32)
4. Patch diagonal blocks: convert `diagonal_blocks` to f32 pairs, `queue.write_buffer` into `matrix_buf` at diagonal offsets
5. Return `Ok(Box::new(session))`

### Wiring in `cda/mod.rs`

The GPU assembly path is only attempted when FCD, substrate, and periodic BC are all absent — the shader implements free-space G only:

```rust
// GPU assembly: only when no FCD, no substrate, no Ewald periodic BC
let gpu_session = if !use_fcd && params.substrate_runtime.is_none() && self.lattice.is_none() {
    backend.assemble_and_create_session(&positions, k, &diagonal_blocks)
} else {
    None
};

let session: Box<dyn MatvecSession> = match gpu_session {
    Some(Ok(s)) => s,
    _ => {
        // Existing path: CPU Rayon assembly + upload
        let matrix = assembly::assemble_interaction_matrix(...);
        let mut s = backend.create_session(3 * n)
            .ok_or(SolverError::ComputeError(...))?;
        s.upload_matrix(&matrix)?;
        s
    }
};
```

`diagonal_blocks` are computed via the new `assembly::compute_diagonal_blocks(dipoles)` before this block. Non-GPU paths (on-the-fly, cached CPU) are completely unaffected.

### Activation guard summary

The GPU assembly shader is only active when ALL of:
- GPU backend present and matrix fits memory budget
- `use_fcd == false`
- `self.lattice.is_none()` (no periodic BC)
- `params.substrate_runtime.is_none()` (no substrate)

Otherwise falls back to CPU `assemble_interaction_matrix`.

### Diagonal patching

The α⁻¹ blocks (3×3 per dipole, CPU computed via `invert_3x3`) are pre-computed before the GPU dispatch and written into the matrix buffer at offsets `(3i)*(3N) + 3j` for i==j. This is a `write_buffer` call of 9·N complex f32 values — O(N), negligible vs O(N²) shader.

## Solver Dispatch Summary (after v0.5.0)

The existing `matrix_memory_budget` gate (default 2 GiB) determines whether the 3N×3N matrix can be cached. The FFT path is valid regardless of the budget (it never stores the full matrix), so it is inserted **before** the budget check:

```
solve_dipoles(N dipoles):
  N ≤ 1000
    → direct LU (faer)

  N > 1000, regular grid, no substrate, no periodic BC, no FCD
    → FftMatvecPlan::try_build()  [FFT matvec, NEW — O(N log N)]
    → GMRES with FftMatvecPlan::matvec

  N > 1000, matrix fits budget (3N)²·16 ≤ matrix_memory_budget:
    GPU available, no FCD, no substrate, no periodic BC
      → assemble_and_create_session()  [GPU shader, NEW]
      → GMRES with GpuSession matvec
    GPU unavailable or FCD/substrate/Ewald present
      → assemble_interaction_matrix (CPU Rayon)
      → GMRES with session matvec (CPU cached)

  N > 1000, matrix exceeds budget:
    → GMRES with matvec_on_the_fly (existing)
```

**FFT takes precedence over GPU** for applicable geometries: FFT is O(N log N) memory and O(N log N) time vs GPU assembly's O(N²) upload and O(N²) shader work. For a regular grid without FCD/substrate/periodic BC, FFT is always preferred. The GPU assembly path covers the remaining cases (non-regular grid, or FCD/substrate/periodic BC present) where the full matrix must be assembled anyway.

GPU assembly and FFT matvec are independent — a future version could combine them for GPU-side FFT convolution, but that is deferred.
