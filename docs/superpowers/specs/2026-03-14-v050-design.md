## Feature 1 — 1D Bessel Spectral Sum

### Problem

`EwaldGreens::recip_space()` returns a zero tensor for `LatticeSpec::Chain` (1D periodic BC). The real-space direct sum with 20 shells (~2% accuracy) is the only contribution. For unit cells where dipoles have non-zero transverse separation ρ > 0 (e.g. a 1D periodic dimer or helical chain), the reciprocal-space spectral sum is needed for correct far-field behaviour.

### Physics

For a 1D chain periodic in x with Bloch factor exp(ik_Bx·x) and period a₁, Poisson summation gives:

```
G_recip(Δr) = (1/2πa₁) Σ_m  exp(i G_m Δx) · T(G_m, ρ, κ_m)
```

where:
- G_m = k_Bx + 2πm/a₁ (reciprocal lattice vector)
- κ_m = csqrt_positive_imag(G_m² − k²)  (Im ≥ 0 branch — already in codebase)
- ρ = √(Δy² + Δz²)  (transverse separation)
- T is a 3×3 tensor assembled from K₀(κ_m ρ) and K₁(κ_m ρ)/ρ via ∇∇ applied to the scalar sum

The tensor components follow from applying ∂ᵢ∂ⱼ to the scalar spectral sum:
- Longitudinal (xx): −G_m² terms + k² contribution
- Transverse (yy, zz) and mixed (xy, xz, yz): K₁(κ_m ρ)/ρ and K₀ terms weighted by Δy/ρ, Δz/ρ

### Mode dispatch

| Mode | κ_m | Bessel functions used |
|------|-----|----------------------|
| Evanescent (G_m > k) | real positive β | K₀(βρ), K₁(βρ) — real-valued |
| Propagating (G_m < k) | purely imaginary iα (α > 0) | K₀(iαρ) = −(π/2)(Y₀(αρ) + i J₀(αρ)); K₁ similarly via Y₁/J₁ |

J₀, J₁ already available in `sommerfeld.rs` — will be extracted to a shared `bessel.rs` or re-implemented locally in `ewald.rs`.

### ρ = 0 guard

For a strictly collinear 1D chain (all dipoles at y = z = 0), ρ = 0 and K₀(0) diverges. The divergence cancels with the real-space sum, but computing them separately is numerically unstable. Guard: `if ρ < 1e-10 { return zero_tensor; }`. The 20-shell real-space sum is the full answer in this case.

### New private functions (all in `ewald.rs`)


### Implementation notes

- Shell count M = ⌈k·a₁/2π⌉ + 10 each side — ensures |κ_m|·ρ ≥ 10 for outermost evanescent term, giving exponential suppression < e⁻¹⁰
- `bessel_k0`/`bessel_k1`: piecewise A&S 9.8.5–9.8.8 polynomial rational approximations, accuracy < 1.5×10⁻⁷ (consistent with existing `erfc_real`)
- `bessel_y0`/`bessel_y1`: piecewise A&S 9.4.1–9.4.6, same accuracy target

## Feature 2 — FFT-Accelerated Matvec (Block-Toeplitz)

### Problem

`matvec_on_the_fly` is O(N²) per GMRES iteration. For large nanoparticles on a regular cubic grid (the typical output of `discretise_primitive`), the interaction matrix is 3-level block-Toeplitz — the off-diagonal blocks depend only on the displacement r_i − r_j. A 3D FFT convolution reduces the matvec to O(N log N).

### Activation condition

All four conditions must hold:
1. `N > iterative_threshold` (GMRES path active)
2. `solver.lattice.is_none()` — no periodic boundary conditions
3. `params.substrate_runtime.is_none()` — no substrate (breaks translation invariance in z)
4. `detect_regular_grid(dipoles, cell_size).is_some()` — dipoles on a regular cubic grid

Falls back to `matvec_on_the_fly` silently if any condition fails.

### Regular lattice detection

```rust
fn detect_regular_grid(dipoles: &[Dipole], cell_size: f64)
    -> Option<(usize, usize, usize)>
```

Algorithm:
1. Compute bounding box; grid dimensions `Nx = (x_max − x_min)/cell_size + 1` (rounded)
2. For each dipole position, check `|(pos[α] − box_min[α]) / cell_size − round(...)| < 1e-6`
3. Return `Some((Nx, Ny, Nz))` only if all dipoles pass; `None` otherwise

### `FftMatvecPlan` struct (new file `cda/fft_matvec.rs`)

```rust
pub struct FftMatvecPlan {
    nx: usize, ny: usize, nz: usize,
    /// Flat index for each dipole in the (2Nx × 2Ny × 2Nz) padded grid
    grid_index: Vec<usize>,
    /// Precomputed FFT of 9 kernel components G_αβ (α,β ∈ {0,1,2})
    /// Shape: 9 × (2Nx · 2Ny · 2Nz)
    kernel_fft: Vec<Vec<Complex64>>,
    fft_len: usize,   // 2Nx · 2Ny · 2Nz
}
```

```rust
impl FftMatvecPlan {
    pub fn try_build(dipoles: &[Dipole], cell_size: f64, k: f64)
        -> Option<Self>;
    pub fn matvec(&self, dipoles: &[Dipole], x: ArrayView1<Complex64>)
        -> Array1<Complex64>;
}
```

### Kernel precomputation

For each component pair (α, β) ∈ 0..9:
1. Allocate a `(2Nx × 2Ny × 2Nz)` complex buffer, initialised to zero
2. For each displacement (Δl, Δm, Δn) with (Δl,Δm,Δn) ≠ (0,0,0):
   - `r = [Δl·d, Δm·d, Δn·d]`; compute G_αβ(r) via `dyadic_greens_tensor`
   - Write to circulant position: `(Δl % 2Nx, Δm % 2Ny, Δn % 2Nz)` using Rust `rem_euclid`
3. Run 3D FFT (via three passes of 1D FFTs along x, y, z axes using `rustfft`) and store

The G(0,0,0) = 0 entry (self-interaction excluded by circulant placement — zero-initialised buffer).

### Per-matvec computation

1. Diagonal: `y[3i..3i+3] += α_i⁻¹ · x[3i..3i+3]`
2. For β ∈ {0,1,2}:
   a. Scatter `x_β` components onto the `(2Nx × 2Ny × 2Nz)` padded grid
   b. Forward FFT3
   c. For α ∈ {0,1,2}: pointwise multiply with `kernel_fft[α*3+β]`, add to accumulator
3. Inverse FFT3 for each α accumulator; gather `y_α` from grid positions


## Feature 3 — GPU Matrix Assembly Shader

### Problem

Currently, the full 3N×3N interaction matrix is assembled CPU-side by Rayon (O(N²) Green's function evaluations) and then uploaded to the GPU. For large N, the CPU assembly time dominates. A GPU compute shader can evaluate all (i,j) Green's tensor blocks in parallel, eliminating the CPU bottleneck and the PCIe upload of the fully-assembled matrix.

### Shader: `matrix_assemble.wgsl`

Dispatch: `⌈3N/16⌉ × ⌈3N/16⌉` workgroups, 16×16 threads per group — one thread per (row, col) in the 3N×3N matrix.

Per-thread:
1. `i = row / 3`, `j = col / 3`, `α = row % 3`, `β = col % 3`
2. If `i == j`: write `(0.0, 0.0)` (diagonal block left as zero; α⁻¹ patched separately)
3. Otherwise: compute `R = positions[i] - positions[j]`, evaluate `G_αβ(R, k)` in f32, write to `matrix[row * dim + col]`

Bindings:
```wgsl
@group(0) @binding(0) var<storage, read>       positions : array<vec4<f32>>;
@group(0) @binding(1) var<storage, read_write>  matrix   : array<vec2<f32>>;
@group(0) @binding(2) var<uniform>              params   : Params;
// Params { dim: u32, n_dipoles: u32, k_re: f32, k_im: f32 }
```

The free-space dyadic G expression is identical to `dyadic_greens_tensor` in greens.rs, translated to f32 WGSL. The f32 precision floor (~1e-7) is consistent with the existing GPU matvec shader.

### New trait method on `ComputeBackend`

Default implementation returns `None` (CPU backend unchanged):

```rust
fn assemble_and_create_session(
    &self,
    positions: &[[f64; 3]],
    k: Complex64,
    diagonal_blocks: &[[[Complex64; 3]; 3]],  // α⁻¹ for each dipole
) -> Option<Result<Box<dyn MatvecSession>, ComputeError>> {
    None
}
```

GPU implementation:
1. Convert positions and diagonal blocks to f32
2. Allocate positions buffer on GPU, upload
3. Dispatch `matrix_assemble.wgsl`
4. Patch diagonal blocks via `queue.write_buffer` (9·N f32 pairs, in-place into the matrix buffer)
5. Return `GpuSession` (matrix buffer now contains the full A matrix)

### Wiring in `cda/mod.rs`

In the existing GPU-session branch, before the current `assemble_interaction_matrix` + `upload_matrix` path:

```rust
let session = backend
    .assemble_and_create_session(&positions, k, &diagonal_blocks)
    .unwrap_or_else(|| {
        // existing CPU assemble + upload
        let matrix = assembly::assemble_interaction_matrix(...);
        session.upload_matrix(&matrix)
    });
```

Non-GPU paths are completely unaffected.

### Diagonal patching

The α⁻¹ blocks (3×3 per dipole, CPU computed via `invert_3x3`) are pre-computed before the GPU dispatch and written into the matrix buffer at offsets `(3i)*(3N) + 3j` for i==j. This is a `write_buffer` call of 9·N complex f32 values — O(N), negligible vs O(N²) shader.

## Solver Dispatch Summary (after v0.5.0)

```
solve_dipoles(N dipoles):
  N ≤ 1000 → direct LU (faer)
  N > 1000, GPU available, matrix fits budget:
    → assemble_and_create_session() [GPU shader, NEW]
    → GMRES with GpuSession matvec
  N > 1000, regular grid, no substrate, no periodic BC:
    → FftMatvecPlan::try_build() [FFT matvec, NEW]
    → GMRES with FftMatvecPlan matvec
  N > 1000, otherwise:
    → GMRES with matvec_on_the_fly (existing)
```

Note: GPU assembly and FFT matvec are independent — a future version could combine them (GPU + FFT), but for v0.5.0 the GPU path takes precedence when the GPU is available.
